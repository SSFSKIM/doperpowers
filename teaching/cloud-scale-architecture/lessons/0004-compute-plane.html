Lesson 4 · warm state는 디스크 문제다

Compute plane — gVisor per-run pods

밀리초 스냅샷 경주는 우리 문제가 아니다. run 길이가 그렇게 만든다 — 그리고 격리를 사는 것은 hypervisor가 아니라 blast-radius 통제다.

이 레슨이 끝나면: Firecracker 메모리 스냅샷이 4–28ms restore에도 왜 기각됐는지 "run 길이 상대성" 한 문장으로 말하고, 프롬프트 주입된 에이전트에 대한 필수 통제 셋이 왜 hypervisor가 아닌지 대고, warm state를 디스크 사다리로 그리고, 1,000 concurrent가 왜 봉투가 아니라 곡선 위의 한 점인지 설명할 수 있다.

1. 출발점: run 길이가 스냅샷 경주를 무의미하게 만든다

샌드박스 업계의 군비 경쟁은 restore latency다: Firecracker는 메모리 스냅샷 restore를 4ms(연구)~28ms(실측 writeup)까지 낮췄고, Morph는 VM 통째로 <250ms에 branch한다. 이 숫자에 끌려가면 답은 "microVM 메모리 스냅샷"이 된다. 하지만 그 답은 초 단위 인터랙티브 REPL 샌드박스(E2B의 사업 전체)를 위한 것이지, 우리처럼 분–시간 단위로 도는 build/test run을 위한 것이 아니다.

결정적 재구성: 5초짜리 fresh gVisor start는 30분 run에 대해 0.3% 오버헤드다. 밀리초를 아끼려고 지불하는 대가가 그 아낌보다 크다면, 경주에 참가할 이유가 없다. 그 대가는 가설이 아니라 Firecracker 프로젝트 자신의 문서에 적혀 있다:

resume 결함무슨 일이 벌어지나
Entropy/identity한 스냅샷을 N개 클론으로 restore하면 guest 엔트로피 풀·캐시된 난수·토큰·UUID가 복제된다. Linux 5.18+ VMGenID reseed 후에도 "unique identifiers, cached random numbers, cryptographic tokens는 여전히 복제된다."
Clockguest wall-clock이 스냅샷 생성 시각에서 재개 — 수동으로 고쳐야 하고, 고치면 시계가 "갑자기 점프"한다.
Network/TCP"guest network connectivity is not guaranteed to be preserved after resume"; 열린 vsock 연결은 닫힌다.
The production breakage story Fly.io는 이 셋을 프로덕션에서 전부 맞았다: resume된 머신이 JWT 검증·cron·cache TTL·TLS 검증을 한동안 깨뜨렸고, 256MB 머신 resume이 인시던트 중 30초 이상 걸렸으며, 스냅샷은 배포·호스트 마이그레이션 때마다 폐기되고 cold start로 폴백했다. fresh-boot-per-run 설계는 이 세금을 구조적으로 내지 않는다 — 매번 새 커널, 새 시계, 새 엔트로피. — Firecracker snapshot-support.md; Fly.io suspend/resume 인시던트; 연구 문서 §7.3, R2 §Q1

같은 논리로 CRIU(컨테이너 checkpoint/restore)도 탈락한다: 시간이 할당 가상 메모리에 비례해 멀티-GB에서 30–60초 급이고, 연구 문헌이 "too expensive for speculative agent execution"이라 못 박았다. 결론은 한 줄이다 — 메모리 스냅샷은 건너뛴다.

2. 위협 모델의 정직함: 격리를 사는 것은 hypervisor가 아니다

"per-run VM"을 정당화하려면 위협을 정직하게 적어야 한다. 우리 테넌트는 적대적이지 않다(팀은 격리하되 서로를 공격하지 않는다). 실제 공격자는 repo 콘텐츠·PR·의존성에 올라탄 indirect prompt injection이, 완전한 권한을 가진 에이전트를 조종하는 것이다. NVIDIA 샌드박싱 가이드와 Microsoft "prompts become shells" 연구가 지목하는 필수 통제는 hypervisor가 아니라 blast-radius 통제다:

필수 통제 (mandatory)무엇을 막나
egress deny-by-default + 도메인 allowlist proxy탈취된 에이전트가 데이터를 밖으로 빼내거나 C2에 연결하는 것
workspace-scoped writes워크스페이스 밖 파일시스템 오염
per-run scoped credentials토큰 남용 — 자격증명을 애초에 run 범위로 좁혀 발급

NVIDIA의 표현 그대로: "egress를 기본 차단하는 network policy + scoped IAM binding과 결합된 샌드박스는 제 일을 하도록 설득당하지 않는다." 이 셋이 있으면 hypervisor는 격리의 주 통제가 아니다. 그럼 gVisor는 왜 쓰나? 잔여 리스크 하나 — 커널 0-day를 통한 컨테이너 탈출 — 때문이다. gVisor(runsc)는 syscall을 유저스페이스 커널로 가로채 커널 공격면을 VM 비용 없이 줄인다.

The calibration point Anthropic이 Claude Code용으로 오픈소스화한 sandbox-runtime은 bubblewrap + 도메인-allowlist egress proxy — 컨테이너도 VM도 전혀 없다. 가장 많이 배포된 코딩 에이전트의 벤더가, 정확히 우리 워크로드 클래스에 대해 OS-레벨 namespace + egress 필터링을 충분하다고 판정한 것이다. 우리가 gVisor를 쓰는 건 그보다 한 칸 위이지, 두려움 때문에 두 칸 위인 Kata/microVM으로 가는 게 아니다. — Anthropic sandbox-runtime; NVIDIA sandboxing guidance; 연구 문서 §7.3

비용 감각을 숫자로: gVisor 오버헤드는 Google이 Ant에서 실측한 프로덕션 수치로 앱의 70%가 <1%, 추가 25%가 <3%. 최악은 syscall/파일시스템 무거운 build 워크로드로 10–30%. 반면 Kata는 2026 벤치마크에서 syscall-heavy 작업에 ~47% median 오버헤드 (gVisor 18% 대비)이고 bare-metal/nested-virt 노드를 요구한다. Kata/separate-kernel은 하드 컴플라이언스 경계가 있는 repo 클래스에만 쓴다.

3. GKE Agent Sandbox shape: Template → WarmPool → Claim

격리 런타임을 정했으면 오케스트레이션 모양이 필요하다. GKE Agent Sandbox가 정확히 우리 설계의 first-party 관리형 버전이고(GCP에 안착하면 제품 자체를 채택), 우리는 그 모양을 self-host한다:

핵심 semantics 세 가지, 전부 §2.2에서 settled된 축이다:

  • readyz-as-claim. 원하는 상태(desired state)는 idle 워커 수(readyReplicas)를 센다. claim이 워커를 ready 집합에서 빼면 컨트롤러가 backfill. readiness가 claim 신호를 겸하므로(/readyz가 busy 동안 503), 아무 generic orchestrator나 scale-down·rolling update를 하면서도 busy 워커를 절대 죽이지 않는다.
  • exit-0-and-replace. 끝난 워커는 종료하고 fresh로 교체 — cross-ticket 오염이 정리 규율이 아니라 구조적으로 방지된다. review-bounce 창을 위한 짧은 idle-grace만 둔다(같은 워커에 fix-up이 돌아올 경우 대비).
  • outbound-only. 워커는 dispatch plane으로 장수명 HTTPS를 바깥으로만 건다. 워커 호스트에 inbound 포트 없음. 호스트 추가 = 커맨드 하나 + pool 토큰 하나. 호스트가 클라우드·온프렘 간 fungible해진다.

그리고 우리 고유의 twist — species-split pools. reserved non-spoofable 라벨(repo= 자동 파생, pool= 명명)로 implementer와 reviewer 풀을 분리한다. 리뷰어 샌드박스는 implementer에서 상속하지 않고 fresh하며, read-only repo mount + 전체 build/test 실행, push 자격증명 없음. read-only 워커 클래스는 돈을 아끼는 동시에 적대적 독립성을 강화한다.

4. Warm state는 디스크 문제다 — 메모리가 아니라

메모리 스냅샷을 버렸으니 warm state(repo + deps + build cache, 멀티-GB)를 호스트에 어떻게 빨리 올리나? 답은 디스크 사다리다:

단무엇왜
1pinned toolchain image (크면 SOCI-indexed)SOCI는 pull 시간이 이미지 크기에 거의 무관 — 2.5GB를 ~2.8초에. startup이 이미지 크기가 아니라 working set에 비례.
2post-install workspace snapshot = content-addressed 디스크 artifact, node-local NVMe cache에 (Blacksmith 패턴), object storage에서 backfillFirecracker의 clock/RNG/TCP 세금 없이 Cursor의 "70% faster" 경제성을 얻음.
3run마다 overlayfs clonecache hit이면 sub-second, backfill이어도 초 단위.

왜 network storage가 아니라 node-local NVMe인가? 이것이 substrate 전체를 가르는 결정적 숫자다:

The decisive number 2TB io2 EBS를 64k IOPS로 프로비저닝 ≈ $3,850/월. >1M IOPS를 내는 2TB 소비자용 NVMe ≈ <$200 한 번. CI-급 워크로드는 IOPS-bound이고 network volume은 진다 — RunsOn/Blacksmith/Depot 벤치마크가 전부 수렴한다. 이것이 Cursor의 "단일 호스트 천장은 CPU/RAM이 아니라 disk I/O"라는 발견을 독립적으로 뒷받침한다. — 연구 문서 §7.3, R2 §Q3

드리프트는 이 사다리 위에서 탐지가 아니라 구성으로 불가능해진다: snapshot key = hash(env-spec) ⊕ hash(lockfiles) ⊕ toolchain-image digest. 정확 일치만 restore하고, hit이면 certification 커맨드를 smoke probe로 돌린다(인증 게이트가 공짜 드리프트 탐지기를 겸함). 이 환경 계층은 Lesson 5의 주제다 — 여기서는 warm state가 디스크 artifact라는 것만 붙든다.

5. 용량 산수, 그리고 SaaS 기각

중간값 2 vCPU / 8GB / ~30GB NVMe scratch per run. 코딩 에이전트는 wall-clock 대부분을 LLM 토큰 대기로 놀기 때문에 CPU를 2–4× oversubscribe해도 안전하다(Cursor: CPU/RAM은 천장이 된 적 없다). 그래서:

  • RAM이 packing 제약, disk I/O가 performance 제약. 8GB/run + 여유 → ~80–110 runs/host RAM-packed.
  • I/O 검산: ~⅓의 run이 active build burst(300–500MB/s each)면 host당 ~10–17GB/s — 딱 호스트 천장이고 Cursor의 "many GB/s"와 일치. 즉 ~100/host는 NVMe scratch가 있어야만 현실이고, network volume이면 ~10–20/host로 붕괴한다(약 20× 차이).

여기서 가장 흔한 오독을 못 박아야 한다. 1,000 concurrent는 봉투가 아니라 곡선 위의 한 점(anchor)이다. 관계식은:

concurrency = start rate × mean run duration
anchorstart ratemean runconcurrencyfleet
A2 (mid)1.4/sec (~5k/hr)~20 min~1,000–1,700~12–16 hosts (1,000점); band 상단 1,700이면 ~28
A3 (top)6/sec (≈21.6k/hr)60 min~20,000선형 확장 (spec: ~230–330 hosts)

같은 5k runs/hr도 run이 12분이면 동시 1,000, 1시간이면 동시 5,000이다 — per-hour throughput를 concurrency와 절대 혼용하지 않는다. A2의 1,000-concurrent 점은 작고 지루한 fleet(12–16 호스트)이지만, 6 starts/sec에 hour-long run이면 같은 dispatch 봉투가 concurrency ~20,000에 닿고 fleet은 선형으로 커진다. 그래서 per-host density는 fleet 구매 전에 spike S1(1-host load test)로 먼저 검증한다 — 공개된 primary-source per-host 밀도 숫자가 어디에도 없기 때문이다.

Build vs buy, 숫자로 1,000 sustained concurrent runs에서 usage-priced SaaS 샌드박스는 ~$73k/월(E2B/Daytona rates)에서 ~$104k/월 (Modal — physical core당 과금 ≈ 2 vCPU라 ~1.4×)다. self-hosted bare metal은 ~$5–15k/월. 게다가 SaaS는 다른 문제(초 단위 인터랙티브 REPL)에 최적화돼 있지, 멀티-GB repo 상태로 시간 도는 build/test run이 아니다. E2B만이 오픈 self-host 스택을 갖지만 그마저 Nomad/Consul/KVM estate를 요구한다 — 그 지점이면 k8s+gVisor가 덜 exotic하다. SaaS는 substrate로 기각. — 연구 문서 §7.3, R2 §Q2
