Lesson 2 · 전이를 서버가 쥔다

Board Service 해부

티켓 상태의 SSOT는 왜 SaaS 트래커가 될 수 없었나 — 그리고 한 문장짜리 conditional UPDATE가 어떻게 합법성 검사와 claim 락을 동시에 해내는가.

이 레슨이 끝나면: GitHub도 Linear도 왜 SSOT가 못 됐는지 각각의 결정적 실격 사유를 숫자로 말하고, conditional UPDATE가 Kubernetes API 서버 패턴과 같은 이유를 설명하며, Linear 미러가 "진실이 아니라 이벤트"라는 규칙이 왜 있는지 남에게 설명할 수 있다.

1. 두 SaaS 트래커가 각각 다른 이유로 탈락한다

레슨 1에서 보드는 무엇의 진실이라고 했다. 그 진실을 어디에 담느냐가 이 레슨의 문제다. 후보는 셋이었다: GitHub 이슈(오늘의 구현), Linear(명명된 후보), 그리고 우리가 직접 소유하는 얇은 Postgres 서비스. 작업 부하는 프로젝트당 1,000 runs/hr × ~10 board ops = 10,000 ops/hr, 그중 약 60%가 쓰기 → ~6,000 writes/hr, 디스패치 웨이브라 평균보다 훨씬 버스티하다 [R §7.1].

핵심은, 두 트래커가 서로 다른 벽에 먼저 부딪힌다는 것이다:

후보먼저 깨지는 한도우리 부하 대비판정
GitHub (앱 1개 설치)콘텐츠 생성 2차 한도 500/hr · 80/min (per actor)~6,000 writes/hr → 12배 초과1차 한도(5,000/hr) 전에 콘텐츠 생성에서 붕괴
Linear (OAuth actor 1개)5,000 req/hr per actor10,000 ops/hr → ~2배 초과 (쓰기만 ~1.2배)근소 초과 + 전이 강제·CAS 자체가 없음
얇은 Postgres 서비스~3 TPS/project vs 단일 노드 수천 TPS≥100배 여유처리량과 정확성이 둘 다 닫히는 유일한 옵션

GitHub의 함정은 한도 자체보다 숨은 공유 버킷이다: 2차 한도는 한 앱의 여러 토큰, 또는 같은 IP의 트래픽이 하나의 버킷을 공유할 수 있고, GitHub은 이 버킷팅을 공개하지 않는다. 그래서 "프로젝트당 앱 12개로 샤딩"은 2차 버킷을 곱해준다는 보장이 없다. 자기 SSOT를 벤더의 abuse 시스템에 맞서 샤딩하는 것 자체가 적대적 자세다 [R §7.1, r2 §1.2].

absence-of-evidence 라는 결정타 Linear의 issueUpdate(input: { stateId })는 팀에 속한 어떤 stateId든 그대로 받는다. 서버측 전이 제한 기능도, compare-and-swap(조건부 업데이트) 프리미티브도 문서·체인지로그· 스키마 어디에서도 발견되지 않았다 — Jira·Azure DevOps·Plane이 자기 것을 광고하는 것과 대조된다. 이건 "부재의 증거"다: 공식 "지원하지 않음" 선언조차 없다. 그래서 수천 라이터 간의 claim race는 Linear에서는 닫을 수 없다. 게다가 감사 로그는 90일만 보존 — 영속적 human-answer 기록을 나를 SSOT로는 그것만으로도 실격이다. — Linear rate-limiting, audit-log(90일); 연구 문서 §7.1 · research/2026-07-23-cloud-scale/r2-ticket-store.md §1.1, §4

2. conditional UPDATE — 두 일을 한 문장에

전이 그래프를 어디서 강제할 것인가에는 세 자리가 있었다: 클라이언트측 (오늘 — 모든 워커가 검사), 서버측(API 서비스가 그래프를 소유), reconciliation(웹훅이 불법 쓰기를 되돌림). 클라이언트측은 TOCTOU claim race(읽고-검증하고-쓰는 사이의 창)와 플릿 드리프트 (모든 하네스·버전이 동일한 그래프 사본을 지녀야 하고, 스테일 워커 하나가 보드를 조용히 오염)로 스케일에서 무너진다 [r2 §1.4].

서버측의 핵심은 한 문장이다:

UPDATE ticket SET state='in-progress', owner=$run
  WHERE id=$id AND state='ready-for-agent';
-- rows affected = 0  →  경쟁 패배 (409-style 에러, 클라이언트는 re-fetch)

이 WHERE 절이 합법성 검사와 claim 락을 하나의 원자적 문장으로 합친다. 상태가 ready-for-agent가 아니면 조건이 안 맞아 0행이 갱신되고, 그게 곧 "경쟁에서 졌다"는 신호다 — 별도의 잠금도, 별도의 검사도 없다. 이 배치가 불법 상태를 어떤 리더도 관측할 수 없는 유일한 자리다. 클라이언트측은 불법 상태가 잠깐이라도 라이브가 되고(워커가 그걸 보고 유령 run을 디스패치), reconciliation은 창·순서 무보장·이벤트 손실을 안는다 — legality의 도구가 아니라 convergence(미러 back-edge)의 도구다 [r2 §1.4].

The working reference model 이건 우리가 발명한 패턴이 아니다 — Kubernetes API 서버가 정확히 이것이다: 쓰기는 resourceVersion을 실어 보내고, 스테일 쓰기는 HTTP 409 Conflict로 거부되며 클라이언트는 re-fetch 후 재시도한다. legality는 한 서버에서 동기적으로 강제되고, reconciliation은 오직 수렴만 담당한다(유효성이 아니라). 우리의 WHERE ... AND state=...는 그 409의 SQL 번역이다. — Kubernetes API concepts (resourceVersion + 409); 연구 문서 §7.1 · r2-ticket-store.md §1.4

3. Symphony 전례 — 같은 필요, 내구성 없는 형태

우리와 정확히 같은 문제 클래스를 이미 푼 전례가 있다: OpenAI Symphony(2026년 5월, Linear 기반 코드 오케스트레이션). Symphony v1은 트래커(Linear)를 SSOT로 두고, 전이 강제는 클라이언트측(에이전트)에 맡긴다 — 오늘 우리 패턴 그대로다. 그렇다면 동시성은 어떻게 풀까? 하나의 직렬화 오케스트레이터를 강제해서다: 모든 상태 변이를 한 권위를 통해 직렬화해 중복 디스패치를 막는다. 분산 claiming 프로토콜은 명세되지 않았고, 복구는 DB 없이 재폴링으로 한다 [r2 §1.3].

여기서 정확한 교훈: Symphony는 모양은 검증하고 갭은 실증 한다. "하나의 권위"라는 필요는 우리와 같다 — 하지만 그 필요를 인메모리 단일 프로세스라는 내구성 없는 형태로 담았다. 멀티 오케스트레이터·수천 라이터 스케일로 가면 그 "하나의 직렬화 권위"는 실제 데이터베이스를 가진 실제 서비스가 되어야 한다 — 그게 바로 얇은 board service다. 우리는 Symphony의 필요를 취하고, 그 형태를 내구화한다.

4. Linear 미러 — 진실이 아니라 이벤트

SSOT를 우리 서비스로 옮긴다고 Linear를 버리는 게 아니다. Linear는 100% 인간 대면 surface로 남는다 — 단, 진실이 아니라 미러로. 이게 성립하는 이유는 산수에 있다:

양값왜 중요한가
워커가 만드는 board opsrun당 ~10기계 템포 — 트래커 한도를 터뜨리는 원인
인간이 신경 쓰는 전이그중 ~2–3coalescing의 대상
coalescing 후 트래픽 축소~5–10배샤딩을 논하기 전에 이미 줄어듦
미러의 실제 Linear 쓰기수백/hrOAuth actor 1개 예산 안에 들어감

단방향 sync 워커가 기계 템포 쓰기를 인간 관련 업데이트로 coalesce한다. 반대 방향(back-edge)은 규칙이 엄격하다: Linear에서 인간이 한 행동(park 답변, 우선순위 편집)은 board service로 이벤트로 흘러 들어올 뿐, 진실이 아니다. board service는 여전히 유일한 기록 라이터다 [R §7.1, r2 §3].

PR automation 위험 — 제2의 라이터 Linear의 자체 GitHub 통합은 팀별 상태 자동화를 돈다: PR merged → Done. 이건 조율되지 않은 제2의 티켓 상태 라이터다. 우리 의미층은 in-review → landed를 리뷰 워커의 tiered merge authority로만 넘긴다 — Linear의 "merged → Done"이 그 권한과 정면으로 싸운다. 게다가 여러 PR이 한 이슈에 걸리면 자동화는 마지막 PR에서만 발화 — 분해된 child-ticket 흐름에는 틀린 동작. 팀별로 이 자동화를 끈다. — Linear GitHub integration; 연구 문서 §7.1 · r2-ticket-store.md §1.5

5. 보드가 닿지 않을 때 — fail closed, 그리고 영속 기록

board service가 SSOT라면, 보드가 unreachable일 때 워커가 뭘 하느냐가 중요해진다. 규칙은 fail closed다: 전이를 쓸 수 없는 워커는 자기 로컬 스텝을 끝내고, 산출물을 session store에 내구적으로 park하고, 멈춘다 — 상태 경계를 넘어 가정으로 전진하지 않는다. 코드 호스트 상태를 경계 너머로 쓰지 않고, lease 만료가 그 run을 park한다 (Acceptance 드릴 4: 보드를 파티션시키면 워커가 현재 스텝 완료 → session store에 저장 → 경계에서 정지). 이것이 "불법 전이는 어떤 리더도 관측 불가"라는 §2의 보증을 부분 장애 상황까지 확장한다.

그리고 board service의 history 테이블은 append-only이며 영구 보존된다 — 이 테이블이 곧 내구적 human-answer 기록이다. 이게 Linear를 SSOT에서 탈락시킨 이유 중 하나와 직접 맞닿는다: Linear의 감사 로그는 90일만 보존한다. 영속 기록을 나를 SSOT는 그보다 오래 살아야 한다 [R §7.1].

한 문장으로 board service가 소유하는 것은 어떤 SaaS 트래커도 어떤 값에도 팔지 않는 딱 두 가지다 — 우리 합법-전이 그래프의 서버측 강제와 수천 동시 기계 라이터를 위한 원자적 claim. 의미층이 frozen으로 남는 것은 바로 그 강제가 모든 클라이언트가 제대로 하기를 더는 의존하지 않게 됐기 때문이다.
