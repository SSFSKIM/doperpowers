Lesson 3 · 상주 컨트롤러의 귀환

Dispatch Plane — lease, fencing, 그리고 상주 컨트롤러의 귀환

"누가 이 run을 소유하는가"를 파일이 아니라 한 트랜잭션이 답하게 만드는 프리미티브들 — 그리고 "no resident orchestrator"라는 옛 규칙이 왜 명예롭게 은퇴했는지.

이 레슨이 끝나면: claim이 왜 FOR UPDATE SKIP LOCKED 한 줄이면 충분한지, lease를 run 길이에 맞추면 안 되는 이유, 좀비 프로세스를 fencing token이 어떻게 막는지, 그리고 "상주 오케스트레이터 금지" 규칙이 은퇴하면서도 level-triggered reconcile 독트린은 왜 살아남았는지를 남에게 설명할 수 있다.

1. Claim: 경쟁을 한 줄로 이기기

수천 개의 run이 초당 최대 6개씩 시작되는데, 두 워커가 같은 작업을 동시에 집으면 안 된다. Postgres의 관용구 하나가 이 경쟁을 원자적으로 해결한다:

-- run-queue 테이블에서 잠기지 않은 다음 작업을 원자적으로 집는다
SELECT * FROM run_queue
  WHERE state='pending'
  ORDER BY ...
  FOR UPDATE SKIP LOCKED
  LIMIT 1;

FOR UPDATE는 집은 행을 트랜잭션 락으로 잠그고, SKIP LOCKED는 다른 워커가 이미 잠근 행은 건너뛰고 다음 가용 작업으로 넘어간다. 두 워커가 동시에 폴링해도 서로 다른 행을 집는다 — 락 경합으로 줄 서지 않고, 스핀도 없다. 이것이 Board 레슨에서 본 same-transaction property의 디스패치 쪽 짝이다: 보드 미러 행과 claim 행이 같은 트랜잭션에 커밋되므로 엔진은 제2의 티켓 권위가 구조적으로 될 수 없다.

Fencing token — 좀비를 막는 단조 카운터 claim이 성공할 때마다 그 run의 fence 값이 1 증가한다. 옛 lease를 들고 있던 좀비 프로세스(네트워크가 끊겼다 살아난 워커 등)가 뒤늦게 "작업 완료"를 커밋하려 해도, 이미 fence가 올라간 새 attempt를 그 좀비는 완료할 수 없다 — 오래된 fence는 거부된다. 락은 "지금 누가 집는가"를, fence는 "이 완료가 유효한 세대의 것인가"를 판정한다. 둘은 서로 다른 문제다. — hardbyte PG-queue 벤치마킹의 run_lease 패턴; 연구 문서 §7.2, r2-dispatch-engine.md §Q2·Q5

2. Lease 독트린: run에 맞추지 말 것

가장 자주 저지르는 실수: "run이 2시간 걸리니 lease도 2시간"으로 잡는 것. 이러면 워커가 5분 만에 죽어도 회수까지 두 시간을 기다린다. 올바른 독트린은 정반대다.

규칙이유
lease는 분 단위, run 길이에 절대 맞추지 않음죽은 워커를 초–분 단위로 회수하려면 짧아야 한다
progress-as-heartbeat: 세션 스토어 append가 곧 lease 갱신워커는 매 스텝마다 세션 로그에 append한다 — 그게 자연스러운 살아있음 신호. 별도 하트비트 채널이 필요 없다
만료 = 회수 가능, 작업 손실 아님run은 큐로 돌아가고, 다른 프로세스가 내구 세션 로그에서 resume한다 — 절대 restart가 아니다

이 패턴의 출처는 Anthropic의 self-hosted sandbox 워크큐다. 워커는 reclaim_older_than_ms로 죽은 워커의 작업을 회수하는데, 예제는 reclaim_older_than_ms=2000 — 마지막 활동 후 2초면 lease가 만료된다. 왜 그렇게 짧아도 안전한가? tool 결과를 계속 post하는 것 자체가 liveness를 끊임없이 갱신하기 때문이다. 우리도 JSONL 세션 로그가 매 스텝 progress를 emit하므로 같은 구조가 그대로 성립한다.

reclaim은 resume, restart가 아니다 이 구별이 lease 독트린의 심장이다. run의 정체성은 컴퓨트가 아니라 내구 세션 로그다(Lesson 1). 그러므로 lease가 만료돼 다른 호스트가 run을 회수하면, 그 호스트는 세션 로그를 읽어 중단 지점부터 이어간다 — 처음부터 다시 하지 않는다. 짧은 lease가 "작업을 잃는" 것이 아니라 "컴퓨트만 갈아 끼우는" 이유가 여기 있다.

3. "No resident orchestrator"의 명예로운 은퇴

단일 호스트 시절의 강한 규칙 하나가 있었다: "함수는 수입하되 형태는 거부한다" — cron × 멱등 one-shot 패스로 돌리고, 상주 오케스트레이터는 두지 않는다. 이 규칙은 symphony 비교 §9에서 원칙으로 논증됐고, 반대 이유는 셋이었다:

§9의 상주 서비스 반대 논거내구 상태 + 스테이트리스 HA 컨트롤러가 답하는 방식
in-RAM claim → 단일 인스턴스 강제claim이 Postgres에 산다 → 워커를 수평 확장해도 진실은 하나
재시작하면 재시도 큐가 증발재시도·타임아웃도 내구 저장소에 → 재시작 비용이 0
상주 서비스는 자기 워치독이 필요한 새 실패 도메인스테이트리스 컨트롤러를 HA 쌍 + 리더 선출로 → 죽어도 짝이 이어받음

즉 §9의 반대는 손으로 짠 상주 오케스트레이터에 대한 반대였지, 상주성 자체에 대한 반대가 아니었다. 내구 상태를 깔면 세 반대는 모두 직접 해소된다. 그렇다면 왜 batch cron으로는 부족한가? cron이 정직하게 서빙할 수 없는 것이 딱 두 가지 있다:

  • 초–분 단위 lease reclaim: 죽은 워커를 몇 초 안에 회수하려면 batch 주기로는 늦다.
  • 연속적 admission ordering: 승인 순서 계산 자체는 batch 패스로도 된다(SQL ORDER BY일 뿐). 하지만 그걸 연속적으로 돌리면 그 패스는 이름만 다를 뿐 고빈도 상주 루프다.

그래서 규칙은 이렇게 정정된다: "상주 오케스트레이터 금지"는 공식 은퇴, 그러나 level-triggered reconcile 독트린은 살아남는다 — 죽는 것은 batch 주기뿐. 컨트롤러는 상주하되 스테이트리스이고, 원하는 상태(desired state)는 보드다: "run을 가져야 하는데 없는 티켓들"을 매 패스 계산해 수렴시킨다. symphony §9의 sweep-tick 함수들 (지수 백오프 자동 재개, stall 감지 → kill+park, 보드 기반 취소, dispatch-until-cap, 시작 시 복구)은 같은 함수, 산업용 캐리어로 컨트롤러 로직이 된다.

근거: 상주성 재심의 세 축(내구 저장소가 세 반대를 해소, batch가 못 하는 것은 reclaim + 연속 admission, level-triggered는 생존). 연구 문서 §2.1, §7.2; r2-dispatch-engine.md §Q2·§3.

4. Admission fairness: 작은 테넌트를 굶기지 않기

한 팀이 5천 개 run을 쏟아부으면 다른 팀 티켓이 영영 시작 못 하는가? 막는 장치가 admission fairness다. 두 개의 knob이면 충분하다:

  • per-tenant concurrency cap: 팀/프로젝트당 동시 실행 수 상한 (quota는 동시성에서 건다).
  • lowest-recent-usage-first: 다음에 시작할 run을 고를 때 최근 사용량이 가장 적은 테넌트를 먼저 집는다 (fairness는 admission에서 건다 — 어느 run이 다음에 시작하는가).

이건 Kueue의 Admission Fair Sharing 의미론을 그대로 SQL로 옮긴 것이다. Kueue는 "이미 소비한 리소스를 기준으로 대기 작업을 정렬해, 역사적 사용량이 가장 낮은 큐를 항상 우대"한다 — 시간 기반·물량 기반 starvation을 명시적으로 막도록 설계됐다. 우리에게 이건 어렵지 않다: 부하가 ≤6 starts/sec라, Kueue가 대규모 클러스터에서 푸는 문제를 우리는 수백 줄 SQL로 사소하게 푼다.

5. Worker idempotency protocol: 중복 push를 막는 건 엔진이 아니다

냉정한 사실 하나: 모든 durable-execution 엔진은 side effect에 대해 at-least-once다. Temporal도 예외가 아니다. 정경적 실패는 이렇다 — 워커가 git push를 성공시켰는데 "완료" 이벤트를 기록하기 전에 죽는다. 그러면 엔진은 그 activity를 재시도하고, push가 두 번 일어날 위험이 생긴다. 이 문제는 엔진 선택으로 풀리지 않는다. worker protocol이 푼다:

단계기법
모든 attempt가 안정 run-attempt key 파생ticket-run-fence (Temporal의 workflowRunId+activityId 패턴과 동형)
그 key를 스탬프브랜치명 · PR 본문 · 보드 코멘트에 찍는다
모든 비멱등 side effect를 check-then-act로 감싼다push 전: remote ref가 이미 이 SHA를 담고 있나? / PR 생성 전: 이 attempt key의 PR이 이미 있나?

git이 다행히 친절한 side-effect 표면이라 이게 값싸게 된다: push는 SHA로 멱등이고, PR 생성은 브랜치명으로 dedupe된다. 그러니 크래시 후 재시도가 중복 push를 만드는 것을 막는 것은 엔진이 아니라 이 프로토콜이다 — 어떤 엔진을 골라도 똑같이 우리 몫으로 남는다.

6. 두 개의 미묘한 진실

6a. Cursor의 OCC 위험회피: lease를 에이전트에게 보이지 마라

Cursor는 100개 이상 동시 에이전트 실험에서 두 번 데였다. 먼저 락: "에이전트들이 락을 너무 오래 쥐거나 아예 놓는 걸 잊었다… 스무 에이전트가 두셋의 실효 처리량으로 느려졌다." 그래서 optimistic concurrency(읽기는 공짜, 쓰기는 상태가 바뀌었으면 실패)로 바꿨더니 이번엔 다른 병: "위계가 없어지자 에이전트가 위험회피적이 됐다. 어려운 작업을 피하고 작고 안전한 변경만 했다." 교훈은 명확하다 — 에이전트는 무조건적 배타적 티켓 소유를 경험해야 한다. 이 발견은 agent-vs-agent 공유 상태에 관한 것이지 engine 리스에 관한 것이 아니다. 그러니 우리 제약은: lease는 인프라의 관심사로 남기고, optimistic한 공유 상태 쓰기로 에이전트에게 절대 노출하지 말라. 배타 소유는 이미 semantic layer가 보장한다.

6b. Temporal은 왜 차점자이고, self-hosted는 왜 탈락인가

Cursor·OpenAI Codex·Replit Agent 3가 모두 Temporal로 수렴했는데도 우리는 Postgres-owned dispatch plane을 골랐다. Temporal Cloud는 정직한 차점자로 남는다 — scale이 10×+ 자라거나 lease/fairness 코드 1–3k 줄을 소유하기 싫으면 그때 재심. 오늘 지는 이유는 두 가지: 영구적인 determinism/versioning 세금(빠르게 진화하는 에이전트 루프 코드에 새로 붙는 비용), 그리고 doctrine 리스크(workflow 상태가 ticket 상태를 중력처럼 끌어당김).

Self-hosted Temporal은 달러가 아니라 ops로 탈락했다: 4개 서비스 + Cassandra/ES, 그리고 클러스터 생성 시 고정되어 바꾸려면 전체 클러스터 마이그레이션이 필요한 shard-count 함정(한 엔터프라이즈 증언: 6개월 + 전체 클러스터 이관), 노동 전 월 $2.5–4.5k 인프라. 돈만 봐도 우리 envelope에서 비교가 뒤집힌다 — Temporal Cloud는 ~$50/M actions 기준 mid-band 월 ~$9k, 최상단 discount 전 월 ~$39k이고, 그 지점에선 self-host 인프라-only가 오히려 더 싸다. 그럼에도 탈락 근거는 가격이 아니라 shard 함정 + determinism 세금 + 헤드카운트다.

한 문장으로 claim은 SKIP LOCKED 한 줄로 경쟁을 이기고, fence는 좀비의 뒤늦은 완료를 거부하며, 짧은 lease는 progress로 갱신되어 만료 시 세션 로그에서 resume한다 — 그리고 "상주 금지"는 은퇴하되 level-triggered reconcile은 보드를 desired state로 삼아 살아남는다.
