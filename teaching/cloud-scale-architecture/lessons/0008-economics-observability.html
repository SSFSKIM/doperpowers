Economics & Observability

스케일 사이징의 규율(앵커), 자기 자신을 감시하는 기판(회로차단기), 그리고 인프라가 아니라 토큰이 청구서를 지배한다는 진실 — 그리고 이 코스 전체의 캡스톤.

이 레슨이 끝나면: concurrency를 throughput과 절대 혼용하지 않고 A1/A2/A3 앵커에서 플릿을 사이징하고, 왜 모델 티어가 워커가 아니라 phase로 갈리는지 설명하고, 어떤 게이지가 vanity이고 어떤 신호가 health인지 구분하고, 10개의 cross-cutting 규칙을 화이트보드에 복기할 수 있다.

1. Scale anchors — "1,000 concurrent"는 봉투가 아니라 한 점이다

모든 사이징 문장은 이름 붙은 (start rate × duration) 점에서만 말한다. 스펙은 세 점을 못 박았다:

앵커start ratemean runconcurrencyengine actions/day
A1 (entry)0.3/sec (~1k runs/hr)30 min~500~1M
A2 (mid)1.4/sec (~5k runs/hr)~20 min~1,000–1,700~6M
A3 (top)6/sec (≈21.6k runs/hr)60 min~20,000~26M

핵심 규율은 용어집의 한 줄이다: concurrency = start rate × 평균 run 길이. 처리량과는 절대 같은 물건이 아니다. 같은 5k runs/hr라도 run이 12분이면 동시 1,000, 1시간이면 동시 5,000이 된다. 플릿은 concurrency에 선형으로 붙고 — A2의 1,000-concurrent 점은 NVMe 호스트 ~12–16대, 밴드 상단 1,700에서는 최대 ~28대, A3는 ~230–330대 — board/dispatch 플레인은 비용이 고정·작기 때문에 첫날부터 A3로 사이징한다.

War story — synthesis가 잡은 conflation 초기 사이징은 throughput과 concurrency를 조용히 뒤섞고 있었다. "1,000 concurrent"를 봉투 전체인 양 두고 플릿을 딱 그 한 점에서 잡았는데 — 리뷰 패스가 그것이 (start-rate × duration) 곡선 위의 한 점일 뿐임을 잡아냈다. 봉투의 상단(6 starts/sec × 1시간 run)은 사실 ~20,000 concurrent를 함의했고, 그건 플릿이 20배 틀렸다는 뜻이었다. 그래서 지금 Scale anchors 표가 세 점을 이름으로 못 박고 있다. — 스펙 "Scale anchors" + Surprises & Discoveries; 연구 문서 §7.3

2. 모델 경제학 — 품질은 mix-invariant, 비용은 8배로 벌어진다

Cursor의 swarm 통제 실험(SQLite를 Rust로, 835페이지 매뉴얼만으로)이 준 경제학은 세 개의 숫자로 압축된다. 4시간 컷오프 품질은 모델 mix에 거의 무관하게 73%~85%에 모였는데, 비용은 ~8배로 벌어졌다:

측정숫자함의
전체 run 비용 범위$1,339 (Opus 4.8 hybrid) ~ $10,565 (GPT-5.5 단독)~7.9배 — 품질이 아니라 라우팅이 정한다
워커 플릿 비교 (유사 품질)Composer 2.5 워커 $411 vs GPT-5.5 워커 $9,373~23배 — 토큰 벌크를 어디에 태우냐의 문제
토큰 분포워커가 최소 69%, 대부분 run에서 90%+벌크는 워커에 있다 → 거기서 싼 티어가 값한다

그래서 모델 티어는 워커가 아니라 phase 단위 라우팅 결정이다. frontier 모델은 결정 지점(pre-code gate, decomposition, park/escalate 판단, review verdict)에, 싼 모델은 게이트 통과한 well-scoped 티켓 안의 구현 벌크에. 우리 semantic 레이어는 이 결정 지점들을 이미 격리해 두었으니, 기판은 티어를 per-phase로 라우팅하고 계측하기만 하면 된다.

The Fable-planner lesson Fable 5 planner는 Opus 4.8 planner보다 per-token 가격이 ~2배인데도 planner 청구서는 더 쌌다 — planning 토큰을 훨씬 적게 썼으니까. 그런데 그 run의 워커들이 몇 배 더 많은 토큰을 태웠고, run 전체는 실질적으로 더 비쌌다. 교훈: gate/planner 모델은 자기 청구서가 아니라 total downstream run cost로 평가한다. decomposition 스타일이 워커의 verbosity를 몰고, 가장 싼 planner가 가장 비싼 run을 낳을 수 있다. (per-model prompt profile은 그래서 versioned substrate config — GPT-5.6 Sol은 wording 민감성으로 runaway spiral을 내 실험에서 아예 빠졌다.) — agent-swarms 리포트 §3, §5; 연구 문서 §2.6

3. 토큰 spend가 청구서를 지배한다

인프라 비용은 자기-호스팅 기준 A2에서 ~$5–15k/월이다 — 작고 지루한 숫자. 문제는 fleet tempo의 토큰 spend가 그걸 10–100배 초과할 개연성이 있다는 것이다. 이건 모든 비용 기반 substrate 논증을 다시 프레이밍한다: SaaS 샌드박스가 비싸다는 얘기도, self-host가 싸다는 얘기도, 토큰 청구서 옆에서는 반올림 오차일 수 있다.

그래서 두 가지가 따라온다. 첫째, 이 10–100배는 Fermi 추정이고, spike S3가 그걸 측정으로 대체한다 — phase 티어별로 100개 실제 run을 end-to-end 계량. 둘째, provider rate-limit/quota 아키텍처는 사후 대응이 아니라 first-class design input이다. 그 토큰 tempo에서 provider 쿼터를 어떻게 샤딩·백오프·라우팅할지는 게이트웨이 하나의 세부가 아니라 설계 전제다.

4. Observability — 무엇이 vanity이고 무엇이 health인가

스케일에서 기판은 스스로를 감시해야 한다. Cursor의 swarm spiral — 70,000개 넘는 가속하는 충돌 — 은 사람이 지켜보다가 멈췄다. 그 실패를 자동화하는 게 이 절이다.

4.1 Session-end reason taxonomy

워커마다 종료 이유를 6종 카운터로 찍는다: clean / stream-error / closed / error / timeout / aborted. 이 분류가 infra 실패(→ re-dispatch)와 agent 실패(→ park)를 기계적으로 가른다. 이유 없이는 "실패했으니 다시"와 "실패했으니 사람에게"를 구분할 수 없다.

4.2 Fleet gauges

세 게이지가 플릿의 core: 큐 depth(오토스케일 신호), pending = claimed-not-acked(stall 탐지), workers_polling(liveness). Anthropic self-hosted 샌드박스 프로토콜에서 그대로 가져온 것.

4.3 Contention circuit breakers

per-repo conflict rate, hottest-file contention, PR-rework rate, commit-churn ratio. 여기서 규율은 한 문장이다: volume 메트릭은 vanity, conflict acceleration이 health. 스파이럴은 커밋 수가 많아서가 아니라 충돌이 가속해서 위험하다 — 가속이 그 repo의 dispatch pause를 자동으로 trip시킨다.

Cursor swarmold harness (thrash)new harness
첫 2시간 커밋68,000 (≈70배 pace)—
총 충돌70,000+ (가속 중 pause)4시간 내내 <1,000
hottest file7,771 충돌 · 1,173 agents47

68,000 커밋은 생산성처럼 읽히지만 thrash였다 — vanity 메트릭이 왜 위험한지의 교과서. 같은 telemetry feed 위에 megafile 기계가 얹힌다: 어느 워커든 hotspot 파일을 flag → merge-queue freeze → 자동 등록되는 decomposition 티켓. 그리고 failure-boundary separation: harness/dispatch 로그는 data-bearing 샌드박스 바깥에 살아서, harness 버그 / transport drop / sandbox death를 user-data 환경에 들어가지 않고 구분한다.

5. Code-host load plan — PR 플레인도 같은 한도에 걸린다

GitHub-as-board를 죽인 그 산수가 PR 플레인에도 그대로 적용된다. PR 생성, review comment, verdict comment는 전부 content-generating request이고, 같은 500/hr-per-actor secondary limit(+ 80/min) 아래 있다 — 프로젝트당 A2–A3에서 ~8–14배 초과. mitigation은 순서대로:

순서mitigation효과
1verdict/finding 트래픽을 board service로PR은 transcript가 아니라 link만 나른다 — 대부분의 content request 제거
2review submission 배칭N개 코멘트를 담은 review 1개 = content request 1개
3per-installation actor 샤딩남은 트래픽을 actor 예산으로 분산
4GHES escape hatch조직이 GHES를 돌리면 configurable limit로 탈출

숫자 근거: 연구 문서 §8.11, 스펙 §9. Acceptance drill 9가 이 플레인을 명시적으로 부하 시험한다 — A2-rate에서 actor당 content 요청률을 기록하고 secondary-limit 아래 지속 동작을 보인다.

6. Adoption path & the honest bill

Phase는 calendar가 아니라 관측된 압력에 게이트된다. 각 phase는 독립적으로 가치 있고 되돌리기 싸다:

phase무엇을증명하는 것
P0 planes on one boxboard+dispatch+session store를 한 Postgres에; controller가 cron sweep을 대체; 워커는 아직 로컬 프로세스same-transaction core + worker idempotency at A1
P1 compute planek8s+gVisor 풀을 2–3 NVMe 호스트에; outbound 워커; env registry + certificationdensity spike(S1) 후에야 플릿 사이징
P2 review/merge at tempoN-lens + verifier, auto-land whitelist v1, risk-class별 merge queue, merge-resolver, park-queue SLOfrozen semantic 레이어가 tempo에서 버팀
P3 scale-outA2/A3 앵커로 플릿 성장, canary 티어(auto-land-under-watch), contention breaker를 org-wide로 armedA3에서의 운영

연구가 닫지 못한 unknown은 spike로 남고, 각각 promote/discard 기준을 스스로 선언한다:

spike측정promote / discard

Aggregate ops budget — Bundle A의 정직한 청구서 각 build-vs-adopt 판정은 개별로는 합리적이지만, 합치면 플랫폼 팀이 한꺼번에 소유하는 컴포넌트 목록이 된다: board service + mirror sync · dispatch controller · k8s/gVisor NVMe fleet · environment registry + certification · session store · vault/egress proxy · contention telemetry · merge queue. 이 목록이 phased adoption path의 상시 논거다 — 그 phase가 필요로 하기 전엔 아무것도 짓지 않는다. — 스펙 §8, "Adoption path"; 연구 문서 §8 items 12–13

7. Capstone — 10개 cross-cutting 규칙 (printable recap)

이 표가 코스 전체의 캡스톤 레퍼런스다. 스펙의 불변량이자 연구 §5의 결론 — 어느 레슨의 세부를 잊어도 이 열 줄은 복기할 수 있어야 한다.

#규칙
1Board = ticket 상태의 SSOT, engine = run 소유권의 SSOT, session store = run 히스토리의 SSOT. ticket 상태 충돌 시 board가 이긴다.
2스냅샷(env든 VM이든)은 certificate를 가진 cache일 뿐, 절대 identity가 아니다.
3Compaction/summarization은 durable log를 절대 다시 쓰지 않는다 (요약은 파생 뷰).
4Environment readiness는 dispatcher precondition이지, gate clause가 아니다.
5Mock은 reviewer가 보는 manifest에 선언될 때만 존재한다.
6Credential: scoped보다 structurally unreachable. merge authority와 credential topology는 같은 모양.
7Worker: outbound-only, one-session, exit-and-replace, busy는 절대 kill 안 함.
8bounded한 모든 것(retry·pool·queue)은 bound에서 loud하게 park — 조용히 loop하지도, 작업을 버리지도 않는다.
9Autonomy prompting: 클라우드에서 blocking은 비싸다 — 조용한 대기 대신 loud·routed escalation(park)으로.
10Reviewer 샌드박스는 fresh, 절대 inherited 아님: read-only repo mount + 완전한 build/test 실행, push credential 없음.
