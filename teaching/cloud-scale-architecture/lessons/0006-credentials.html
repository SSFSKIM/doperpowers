Lesson 6 · scoping이 아니라 도달 불가능성

Credentials — structural unreachability

토큰을 좁게 스코프하는 것은 "모델이 이걸론 별거 못 한다"는 가정이고, 그 가정은 모델이 똑똑해질수록 삭는다. 삭지 않는 유일한 자세는 생성 코드가 토큰에 구조적으로 도달할 수 없게 두는 것이다.

이 레슨이 끝나면: "왜 스코핑이 아니라 unreachability인가"를 Anthropic의 논증으로 재구성하고, git 토큰이 에이전트 눈에 한 번도 안 보이는 경로를 그리고, 크리덴셜 토폴로지가 왜 머지 권한 티어와 같은 모양인지, 그리고 이 설계로도 못 닫는 egress 경로가 무엇인지 남에게 설명할 수 있다.

1. 출발점: scoping은 삭는 가정이다

결합된(coupled) 설계에서는 untrusted 생성 코드가 크리덴셜과 같은 컨테이너에서 돌았다. 그래서 공격 조건이 우스울 만큼 낮았다 — Anthropic의 표현으로 "prompt injection이 해야 할 일은 Claude가 자기 환경(environment)을 읽도록 설득하는 것뿐"이었다. 토큰을 훔치면 공격자는 "새 무제한 세션을 띄우고 거기에 일을 위임"할 수 있다.

흔한 방어는 토큰을 좁게 스코프하는 것이다. Anthropic의 핵심 논증은 이걸 정면으로 부정한다: 좁은 스코핑 자체가 "제한된 토큰으로 Claude가 무엇을 못 하는가"에 대한 가정이고, 그 가정은 "Claude is getting increasingly smart" 하기 때문에 시간이 지나면 삭는다(stale-able). 오늘 안전한 스코프가 다음 모델 세대에서도 안전하다는 보장이 없다. 그래서 방어를 토큰의 좁기에 걸지 말고, 토큰이 생성 코드가 도는 곳에서 구조적으로 도달 불가능하다는 사실에 걸어야 한다.

스코핑 vs. unreachability 스코핑은 탐지·대응형 완화다 — "이 토큰으론 이만큼만 된다"는 가정을 계속 재검증해야 한다. structural unreachability는 구성형이다 — 토큰이 애초에 손 닿는 곳에 없으니, 모델이 아무리 똑똑해져도 읽을 대상이 없다. 이건 이 코스의 반복 패턴이다: 드리프트를 탐지하지 말고 구성으로 불가능하게 만들어라(environment key), 엔진의 월권을 규칙으로 막지 말고 같은 트랜잭션으로 물리적으로 막아라(same-transaction property). — 연구 §2.4; 스펙 §5

2. Git: auth를 리소스에 번들한다

가장 싸고 가장 먼저 했어야 할 조치. git push/pull은 토큰이 필요하지만, 에이전트가 그 토큰을 만질 이유는 없다. 그래서:

  • 샌드박스 init 시점에 short-lived·repo-scoped 토큰으로 리포를 clone하고, 그 토큰을 로컬 git remote 설정에 배선한다.
  • 샌드박스 안에서 push/pull은 그냥 된다 — remote가 이미 인증되어 있으니까.
  • 에이전트는 토큰을 한 번도 보지 않는다. 환경 변수에도, 세션 transcript(JSONL)에도 새어 나갈 것이 없다.

Anthropic은 이걸 두 패턴 중 하나로 부르고("auth bundled with the resource"), 나머지 하나가 §5의 vault+proxy다. 연구 문서는 이걸 "cheap at ANY scale — 처음부터 취했어야 한다"고 못 박는다. 단일 호스트에서도 옳았고 플릿에서도 옳다 — 스케일과 무관하게 공짜에 가까운 구조적 이득이라, 규모를 핑계로 미룰 이유가 없었다.

3. Per-claim 크리덴셜: 발급·만료·로테이션

git 외의 크리덴셜(빌드, 환경 접근 등)은 dispatch 시점에 claim별로 새로 발급(mint)된다. 오래 사는 정적 토큰이 아니다:

속성설계
발급 시점dispatch(claim) 때 per-claim으로 mint
수명STS-style ~1시간 만료 + auto-refresh
스코프그 run의 environment로 한정(least-privilege)
빌드 시크릿build step에만 스코프 — 돌아가는 에이전트 환경엔 부재
로테이션mounted 경로의 토큰 파일 re-read로 갱신 — 재시작 없음

로테이션 메커니즘은 Cursor의 self-hosted 워커에서 그대로 가져온 모양이다: 컨트롤러가 short-lived 토큰을 마운트 경로에 얹고 (--auth-token-file), 워커는 reconnect 때 파일을 다시 읽는다 — pod 재시작 없이 토큰이 갱신된다. 오래 사는 플릿은 어떤 정적 크리덴셜보다도 오래 살기 때문에, 로테이션이 워커의 일급 기능이어야 한다. 빌드 시크릿을 build step에만 가두는 것도 Cursor의 dev-environments 문서에서 온 규칙이다("build secrets are scoped to the build step and aren't passed to the running agent's environment").

왜 ~1시간인가 만료가 짧을수록 탈취된 토큰의 유효 창이 좁다. STS-style ~1h + auto-refresh는 "정상 run은 갱신으로 끊김 없이 살고, 탈취 토큰은 한 시간 안에 죽는다"의 균형점이다. 여기에 per-environment egress allowlist가 겹쳐 — 설령 토큰이 있어도 갈 수 있는 목적지가 정해져 있다. — 스펙 §5; 연구 §2.4

4. 토폴로지가 머지 권한 티어를 거울처럼 따른다

이 레슨의 아키텍처적 핵심. 크리덴셜을 "누가 무엇을 들고 있나"의 지도로 보면, 그 지도는 tiered merge authority와 정확히 같은 모양이어야 한다:

주체들고 있는 크리덴셜할 수 있는 것
Implementer runrepo-scoped·short-lived (PR 열기까지)브랜치 push, PR open
Landing path (land worker)merge-capable 크리덴셜PR merge

merge-capable 크리덴셜은 landing path에만 존재한다. implementer는 PR을 열 수는 있어도 절대 merge할 수 없다 — 물려받은 토큰에 merge 권한이 없기 때문이다. 이건 리뷰 규칙("implementer가 자기 걸 머지하면 안 된다")을 정책으로 부탁하는 게 아니라, 크리덴셜 토폴로지로 강제한다. Lesson 1의 same-transaction property가 엔진의 월권을 물리 법칙으로 막았듯이, 여기서는 크리덴셜의 부재가 implementer의 월권을 막는다. 권한 티어와 크리덴셜 토폴로지는 같은 하나의 모양이다.

5. Vault + egress proxy: 왜 이제야 채택인가

내부 서비스(캐시, 레지스트리, 사내 엔드포인트)에 붙는 service credentials는 vault 뒤에 두고, 에이전트는 egress proxy를 통해서만 그 서비스에 닿는다 — proxy가 run 정체성으로 실제 크리덴셜을 vault에서 꺼내 호출한다. 에이전트도 harness도 진짜 크리덴셜을 보지 못한다.

주목할 점은 이 메커니즘이 예전엔 기각됐다는 것이다. FD-6의 기록된 사다리는 fine-grained PAT를 먼저 쓰고, blast radius가 그래도 너무 넓을 때만 broker(vault/proxy)로 올라가는 순서였다. 단일 호스트 스케일에선 그게 옳았다 — 워커 몇 개에 vault+proxy는 과설계다.

사다리가 무너지는 지점 플릿 스케일에서 blast radius는 run 수 × 팀 수로 커진다. 수백 개 자율 run이 여러 팀의 리포를 건드리면 "언젠가 어떤 run은 prompt injection을 당한다"를 가정해야 하고, 그때 PAT-first 사다리의 첫 rung은 이미 너무 넓다. 그래서 사다리의 PAT 단을 건너뛰고 곧장 vault+proxy로 간다. 스케일이 독트린을 바꾼 게 아니라, 예전 기각의 발동 조건(multi-tenant scale)이 실제로 발동했을 뿐이다. — 연구 §2.4, §3 reopen-trigger audit ("FD-6 PAT-first: superseded")

단, 채택 방식은 절제되어 있다: 별도 bespoke 머신을 짓지 않고 엔터프라이즈의 기존 secrets manager에 대고 구현하며, run 정체성으로 keyed한다. "새 인프라를 세우자"가 아니라 "이미 있는 vault를 run 단위로 쓰자"다.

6. 정직한 경계, 그리고 acceptance drill 6

이 모든 data-plane 설계로도 못 닫는 egress 경로가 하나 있다: 외부 model API를 쓰는 순간, model context window 자체가 egress 경로다. 모델이 추론 중에 읽는 것은 어딘가로 전송된다.

이건 우리가 지어낸 걱정이 아니라 Cursor 자신의 문서가 인정하는 사실이다: "self-hosted" 모드에서조차 "file chunks the model reads during inference are sent to Cursor". 마케팅 문구 "코드는 당신의 환경을 떠나지 않는다"는 대량 저장·실행에 관한 말이지 토큰 스트림에 관한 말이 아니다. 그래서 이 아키텍처를 베끼는 누구든 이 경계를 정직하게 문서화해야 한다 — data-plane 설계는 이 경로를 닫지 못한다.

Acceptance drill 6 — red-team으로 검증되는 것 샌드박스 안의 어떤 프로세스도 git 토큰을 읽을 수 없고(토큰은 remote config에 short-lived 크리덴셜로만 존재), non-allowlisted 도메인에 닿을 수 없고, PR을 merge할 수 없다 — merge는 landing path만 한다. 실제 red-team prompt-injection run이 확인하는 결론: blast radius = board comments + 자기 브랜치, 그 이상 아무것도 없음. 이게 §1의 "scoping이 아니라 unreachability" 논증이 통과했다는 증거다. — 스펙 §Acceptance drill 6
