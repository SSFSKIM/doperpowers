Lesson 7 · 권한의 4층과 verifier

Review & Merge

머지 큐는 스케일에서 어디까지 증명됐고, 누가 인간 없이 랜딩할 수 있으며, 발견의 홍수가 인간 티어를 DoS하지 않게 막는 부품은 무엇인가.

이 레슨이 끝나면: 공개된 머지 큐 용량 계급을 순서대로 말하고, TAP와 SubmitQueue가 각각 무엇을 포기·유지하는지, 머지 권한 4층이 어디서 갈리는지, 그리고 "verifier stage 없이 aggressive 리뷰는 성립하지 않는다"가 왜 산수의 문제인지 남에게 설명할 수 있다.

1. 용량 계급 — 어디까지 공개 증명이 있나

머지 큐 처리량은 취향이 아니라 공개된 하드 계급이다. 같은 단일 리포에서 지속 가능한 랜딩 레이트를 낮은 것부터 세우면 이렇게 된다:

시스템지속 레이트 (per repo)메커니즘실패 시 의미
bors (Rust)~10/day (serial)완전 직렬 테스트; 인간이 rollup으로 묶음—
GitLab merge trains20 parallel (hard cap)train 당 20 파이프라인 병렬drop + 뒤를 재생성
GitHub merge queueconcurrency ≤100 (호스티드 상한)FIFO, speculation·독립성 분석 없음실패 PR 제거 → 후속 그룹 전부 재구성
Shopify~400/daypredictive branch (speculation-lite), deploy 40/day실패 PR auto-remove
Google TAP>50,000/day (~2,000/hr)batch + 실패 배치 auto-bisect범인만 이분 격리; 개별 검증 포기
드롭-앤-리빌드가 왜 에이전트에게 더 아픈가 GitHub·GitLab의 "실패 PR 제거 후 뒤 그룹 재생성" 비용은 대략 실패율 × 큐 깊이의 제곱에 가깝게 자란다. 이 큐들은 인간이 쓴 PR의 낮은 CI 실패율에 맞춰 튜닝됐는데, 에이전트가 쓴 PR은 CI 실패율이 더 높다 — 같은 호스티드 상한도 우리 부하에선 훨씬 빨리 무너진다. 그래서 호스티드 프리미티브는 저레이트 리포 에만 붙인다. — R2 §Q1; 연구 문서 §7.4; 스펙 §6

우리 목표("리포당 수백 runs/hour, 그중 일부가 랜딩")는 Chromium과 Google 사이 구간이다. 이 구간에서 공개 증명이 있는 방식은 딱 둘 — TAP의 batch+bisect, 또는 SubmitQueue의 speculation+독립성. 호스티드 큐는 이 레이트에 배치된 공개 사례가 없다.

2. TAP vs SubmitQueue — 무엇을 포기하고 무엇을 지키나

두 시스템은 같은 벽(직렬 큐는 이 스케일에서 구조적으로 죽는다)에 부딪혀 정반대 대가를 치른다. TAP는 개별 검증을 포기하고, SubmitQueue는 개별 검증을 지키되 투기 비용을 낸다.

Google TAP (batch + bisect)Uber SubmitQueue (speculation + independence)
핵심 거래변경을 묶어 검증, 실패 배치는 자동 이분으로 범인 격리통과 예측 위에 투기 빌드 트리 + 빌드타깃 독립성 분석
개별 검증포기 — 나쁜 변경은 배치에 잠깐 들어왔다 bisect로 빠짐유지 — 개별 변경이 각자 검증되고 랜딩
지렛대 숫자~11분 평균 submit 대기; >50,000/day~100개 피처 로지스틱 회귀, 97% 정확 통과 예측기
독립성 판정—두 변경이 공통 빌드타깃을 안 건드리면 독립 → 완전 병렬 랜딩
성과단일 리포 5만 변경/일 지속oracle 대비 1.2× 턴어라운드; mainline green 52%→100%

인간의 결정(2026-07-23): 머지 검증 의미는 리포의 RISK CLASS별로 고른다. 저위험 클래스는 batch+bisect(throughput을 위해 개별 검증을 맞바꿈), 고위험 클래스는 SubmitQueue식 speculation으로 개별 검증을 유지. 큐는 자체 flake 정책(retry + quarantine + 무고한 PR을 절대 탓하지 않기)을 소유한다.

merge-resolver가 steady-state인 이유 SubmitQueue 논문의 위험 숫자: 단 16개의 in-flight 변경에서 충돌 확률 ≈ 40%. 우리 목표 동시성은 그보다 훨씬 높다. 즉 중립 충돌 해소 에이전트는 엣지 케이스가 아니라 상시 워커 종(種) 으로 예산 잡아야 한다(→ §6). — R2 §Q1; 연구 문서 §7.4; 스펙 §6

3. 머지 권한 4층

2층(auto-land / human)이 아니라 4층이다. ★가 인간 승인 의미층 수정이다. 핵심 경계는 위쪽 두 층이 아니라 맨 아래 — never-auto는 크기와 무관하다.

티어무엇이 판정하나대상
1. auto-land기계검증 가능한 변경 클래스 화이트리스트 + 클래스별 가드레일 체인Rosie 패턴, Renovate 클래스, TestGen-LLM 가드레일이 커버하는 형태만
★ 2. auto-land-under-watch카나리아 판정 조건부: score → promote / route-to-human / auto-revert (Kayenta 모양)중위험 클래스 — 실패가 빠르게 드러나고 값싸게 되돌릴 수 있는 것
3. human tier인간 승인분류 불가하거나 판정이 인간을 요구하는 것
4. never-auto크기 무관 항상 풀 리뷰schema / data / security / config 변경

분류 불가 ⇒ 오토랜드 불가가 정의다. auto-land-under-watch의 근거는 Kayenta(0–100 점수를 success/marginal/failure로 분류)와 Meta의 2% 카나리아 사다리 + Conveyor(배포 파이프라인의 97%가 인간 개입 0) — 이 규모의 모든 조직이 이름은 뭐라 부르든 실질적으로 갖고 있는 층이다.

4. 오토랜드 전례의 정직함

여기서 우리 설계가 전례를 넘어선다는 걸 정직하게 말해야 한다. 공개된 모든 오토랜드 정책은 기계검증 가능한 change-class 화이트리스트 이지, 한 번도 size heuristic이 아니었다:

정책오토랜드 조건 = 클래스 + 가드레일
Google Rosie / LSC패턴 조합(pattern-conforming) LSC shard만 자동 승인; 나머지는 global approver로
Renovate merge-confidence의존성 클래스 + confidence + ≥3일 age(unpublish 창)
Meta TestGen-LLMbuild + 반복 실행 pass(flake 필터) + 커버리지 상승 → 가드레일 통과분의 73% 인간 수용
전례를 넘어선다 — 그리고 그걸 숨기지 않는다 일반 에이전트 코드를 인간 없이 머지하는 곳은 공개적으로 아무도 없다. Anthropic은 거의 모든 내부 PR에 멀티에이전트 Code Review를 돌리지만 PR을 자동 승인하지 않는다. Google은 AI가 쓴 CL(마이그레이션 워크스트림의 80%)도 엔지니어가 승인한다. OpenAI는 ~100만 줄을 수기 0줄로 냈지만 사람이 머지했다. 우리 auto-land 티어는 이 공개 관행을 넘어선다 — 물러설 이유는 아니지만, 그 안전 근거는 산업 전례가 아니라 우리 가드레일 설계 (change-class 화이트리스트 + verifier stage + canary/revert)에 걸려 있다. — R2 §Q2; 연구 문서 §4 flags 7·10, §7.4; 스펙 §6

5. N-lens review (전부 블라인드)와 verifier stage

★ 리뷰 = 하나의 verdict, N개 탈상관 렌즈, 전부 블라인드. 렌즈는 모델과 증거 식단으로 다양화하지만 모든 렌즈가 구현 블라인드(티켓/스펙/트랜스크립트 미수신)를 유지한다. 탈상관이 곧 recall 배수다. 그러나 transcript-aware 렌즈는 기각됐다 — 그건 오늘의 리뷰어가 닫아 둔 증거 경계를 넘기 때문이다. 바뀌는 건 렌즈의 복수성과 aggregation 단계뿐이다.

그 aggregation이 바로 verifier stage — 발견 → 검증 필터 → 심각도 랭킹. 이건 편의 부품이 아니라 자체 false-escalation SLO를 가진 명명된 load-bearing 컴포넌트다. 두 프론티어 랩과 한 벤더가 독립적으로 같은 모양에 수렴했다:

시스템진화 / 모양공개 숫자
Cursor Bugbot8-pass 다수결 투표 → 공격적 finder 하나 + validator로 붕괴; "restrained"→"aggressive"로 전환 (validator가 FP를 흡수하니까)>2M PRs/month; resolution 52% → "over 70%"
Anthropic Code Review병렬 hunter → 검증 필터 → 심각도 랭킹 → 하나의 고신호 코멘트; 자동 승인 안 함vendor 보고 <1% findings 오탐; substantive 리뷰 16%→54%
DoS 실패 모드 — verifier는 선택이 아니다 HackerOne이 2026년 Internet Bug Bounty를 중단했다 — AI로 증폭된 리포트 홍수 아래에서. 걸러지지 않은 LLM 발견 스트림은 인간 triage 티어를 DoS할 수 있다. "aggressive하게 찾기"가 성립하려면 그 뒤에 validator가 반드시 있어야 한다는 건 취향이 아니라 산수다: verifier가 FP를 흡수하기 때문에 loud한 recall이 감당 가능해진다. (인과 읽기 — validator가 aggressive를 감당케 한다 — 는 Cursor의 공개 진화에서 끌어낸 우리 추론이며, Anthropic의 독립 수렴이 이를 뒷받침한다.) — R2 §Q3; 연구 문서 §4 flag 9, §7.4; 스펙 §6

6. merge-resolver(제3종)와 인간 티어 생존 키트

★ merge-resolver는 공식 인정된 제3종(하이브리드)이다. gate도 아니고 자율 워커도 아니다:

속성merge-resolver
템포머지 큐 tempo로 돈다
pre-code gate없음
자체 권한없음 — 산출물은 항상 기존 리뷰 경로로 재진입
다른 유지보수 작업doc 정합·megafile 분해 등은 auto-registered 보드 티켓으로 흡수

맨 아래 인간 티어는 하루 수십 건의 에스컬레이션에서도 살아남는다 — 단 네 가지가 모두 성립할 때만(전부 공개 관행):

생존 조건근거 숫자
작은 단위<200줄 PR은 승인 ~3× 빠름; 200–400줄은 결함 40%↓
사전 검증 + 심각도 랭킹된 에스컬레이션인간은 raw 출력이 아니라 하나의 고신호 요약을 읽음(Anthropic/Bugbot 모양)
전담 approver 로테이션 (ownership 라우팅)Google global-approver analog — 방송이 아니라 라우팅
first-response SLOGoogle: 1 영업일 max, median <1h
park queue도 같은 대우를 받는다 needs-human / needs-info로 park된 것들도 라우팅·배치·통지·SLO를 똑같이 받는다. park 상태는 뒤에 큐 인프라가 있을 때만 스케일에서 살아남는다 — park는 조용한 대기가 아니라 시끄럽고 라우팅된 에스컬레이션이다. — 연구 문서 §7.4, §4.1; 스펙 §6
